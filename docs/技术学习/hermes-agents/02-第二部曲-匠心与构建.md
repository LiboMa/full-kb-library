# 第二部曲：匠心与构建 —— 把 Hermes Agent 锻造成生产力

> 第一部曲里，我们用 30 行 Python 让 Hermes 3 叫出了 `<tool_call>`，像小孩第一次开口说话一样令人激动。但玩具 Agent 和生产级 Agent 之间，还隔着一整条工业流水线：多工具编排、推理链、结构化输出、记忆、多智能体协作、容错、部署……这一部曲，我们要把 Hermes 从"能跑"推到"能干活"。

---

## 0. 承上启下

第一部曲里我们建立了三个关键认知：

1. Hermes Agents 不是一个框架，而是一种"模型原生模式"——Tool Use 的能力内化在模型权重里；
2. ChatML + `<tools>` XML + `<tool_call>` 是它与外界沟通的协议；
3. GOAP 的 `<scratch_pad>` 是模型"想清楚再行动"的脚手架。

Hello World Agent 只调用了一个 `get_weather` 工具。真实世界的 Agent 通常要在一次对话里，规划调用顺序、并行拉取数据、处理部分失败、把结果结构化写回数据库，甚至协调多个子 Agent。

本篇的主线是把 `NousResearch/Hermes-Function-Calling` 仓库里 `functioncall.py`、`jsonmode.py`、`prompter.py` 这几个文件所体现出的工程模式，翻译成你可以直接套用的代码骨架。我们会用七个小节，由易到难，一次吃透。

## 2.1 多工具编排：让模型当自己的项目经理

### 2.1.1 单工具 vs 多工具的世界观差异

Hello World 里的 Agent 是"一次提问 → 一次调用 → 一次回答"的三拍节奏。一旦工具变多，这个节奏立刻崩溃：用户问"帮我看看苹果公司值不值得买"，模型要同时知道价格、财报、基本面——它必须自己决定先调哪个、能否并行、结果怎么综合。

Hermes 3 原生支持在单次 assistant 回合中输出多个 `<tool_call>` 块，这就是并行调用（parallel function calling）。`functioncall.py` 中 `ModelInference.process_completion_and_tool_calls` 的循环正是围绕这个能力设计的：每一轮把模型吐出的所有 tool_call 解析出来，批量执行，再把所有 tool_response 一次性回注。

### 2.1.2 实战：股票研究助手

我们定义三个工具的 JSON Schema（Hermes 3 期待的 tool schema 与 OpenAI function calling 完全兼容）：

```python
TOOLS = [
    {
        "type": "function",
        "function": {
            "name": "get_stock_fundamentals",
            "description": "获取股票基本面信息：市值、PE、行业等",
            "parameters": {
                "type": "object",
                "properties": {
                    "symbol": {"type": "string", "description": "股票代码，如 AAPL"}
                },
                "required": ["symbol"],
            },
        },
    },
    {
        "type": "function",
        "function": {
            "name": "get_financial_statements",
            "description": "获取最近四个季度的财报（营收/净利/现金流）",
            "parameters": {
                "type": "object",
                "properties": {
                    "symbol": {"type": "string"},
                    "period": {"type": "string", "enum": ["quarterly", "annual"]},
                },
                "required": ["symbol"],
            },
        },
    },
    {
        "type": "function",
        "function": {
            "name": "get_stock_price",
            "description": "获取实时股价及当日涨跌幅",
            "parameters": {
                "type": "object",
                "properties": {"symbol": {"type": "string"}},
                "required": ["symbol"],
            },
        },
    },
]
```

System prompt 的关键段落（prompter.py 模式）：

```text
你是一个严谨的股票研究助手。你有以下工具可用：
<tools>
{tools_json}
</tools>

对每个工具调用使用以下 XML 格式返回：
<tool_call>
{"name": "<function-name>", "arguments": <args-json>}
</tool_call>

如果一个任务需要多个独立工具，请在同一回合输出多个 <tool_call> 块以便并行执行。
```

主循环骨架（基于 functioncall.py 简化）：

```python
import json, re, asyncio
from openai import OpenAI  # 指向本地 vLLM 端点

client = OpenAI(base_url="http://localhost:8000/v1", api_key="EMPTY")
TOOL_CALL_RE = re.compile(r"<tool_call>\s*(\{.*?\})\s*</tool_call>", re.S)

LOCAL_FUNCS = {
    "get_stock_fundamentals": get_stock_fundamentals,
    "get_financial_statements": get_financial_statements,
    "get_stock_price": get_stock_price,
}

async def call_one(call):
    fn = LOCAL_FUNCS[call["name"]]
    return await asyncio.to_thread(fn, **call["arguments"])

async def run_agent(user_msg, max_depth=5):
    messages = [
        {"role": "system", "content": build_system_prompt(TOOLS)},
        {"role": "user", "content": user_msg},
    ]
    for depth in range(max_depth):
        resp = client.chat.completions.create(
            model="NousResearch/Hermes-3-Llama-3.1-8B",
            messages=messages,
            temperature=0.3,
        )
        content = resp.choices[0].message.content
        calls = [json.loads(m) for m in TOOL_CALL_RE.findall(content)]
        if not calls:
            return content  # 最终自然语言答复

        # 并行执行所有 tool_call
        results = await asyncio.gather(*[call_one(c) for c in calls])

        messages.append({"role": "assistant", "content": content})
        for c, r in zip(calls, results):
            messages.append({
                "role": "tool",
                "content": f"<tool_response>{json.dumps({'name': c['name'], 'content': r})}</tool_response>",
            })
    raise RuntimeError("max_depth exceeded")
```

三点值得注意的工程细节：

- `max_depth` 是递归护栏。Hermes 在复杂问题下可能连环调用（A 的结果触发 B），必须给它一个上限防止死循环，推荐 5~8；
- 正则解析 `<tool_call>` 比 JSON 整段解析更鲁棒，模型偶尔会在块外写解释性文字；
- `asyncio.gather` 真正把并行兑现了。对于 IO 密集的财务 API，这一步能把延迟从线性降到常数。

### 2.1.3 Tool 依赖链

有时候工具之间不能并行，例如 "先查某公司 CEO，再搜索该 CEO 的最新动态"。这种依赖链不需要你编码，Hermes 会自然地分两个回合处理——第一回合只调 `get_ceo`，看到 `<tool_response>` 后在第二回合再调 `search_news`。你的主循环只需要保持 `max_depth > 1`，其余交给模型的规划能力。

## 2.2 GOAP 推理框架：让模型先思考再动手

### 2.2.1 scratch_pad 是什么

GOAP = Goal Oriented Action Planning，原本是游戏 AI 里的经典规划算法。Nous Research 把它"prompt 化"，让 Hermes 在一个名为 `<scratch_pad>` 的标签里进行显式的内省：

```text
<scratch_pad>
Goal: 用户想要判断苹果公司的投资价值
Actions:
  1. get_stock_fundamentals("AAPL") — 取 PE、市值
  2. get_financial_statements("AAPL", "quarterly") — 看营收趋势
  3. get_stock_price("AAPL") — 当前点位
  这三个互相独立，可并行。
Observation: (等待工具返回后填写)
Reflection: (评估是否还需要补数据)
</scratch_pad>
```

### 2.2.2 完整 Prompt 模板

把以下段落追加到 system prompt 末尾，Hermes 3 就会在每次 tool_call 前先写 scratch_pad：

```text
在每次输出 <tool_call> 之前，你必须先输出一个 <scratch_pad> 块，结构如下：

<scratch_pad>
Goal: <一句话重述用户真正想要的结果>
Actions: <编号列出你计划调用的函数及理由，标注哪些可以并行>
Observation: <若本回合是在工具返回后，总结收到的信息；否则留空>
Reflection: <评估目标是否达成：如果是，请停止调用工具并给出最终答复；否则说明下一步>
</scratch_pad>

Reflection 判定"目标已达成"后，不要再输出任何 <tool_call>，直接给出自然语言回复。
```

### 2.2.3 GOAP vs ReAct：一张表看清楚

| 维度 | GOAP（Hermes） | ReAct（经典 prompt 范式） |
|------|-------------------|---------------------------|
| 结构 | 四段式：Goal / Actions / Observation / Reflection | 两段式：Thought → Action |
| 规划时机 | 调用前一次性规划多个 Action，支持并行 | 每次只想一步、做一步 |
| 反思机制 | 显式 Reflection 段判断终止 | 靠模型自发停止，容易无限循环 |
| 可解析性 | XML 标签边界清晰，易提取调试 | 纯自然语言，正则难抓 |
| 训练支持 | Hermes 3 原生 SFT 过此格式 | 通用 LLM 的 zero-shot 技巧 |
| 延迟 | 一次思考多步执行，延迟低 | 一步一思考，轮次多延迟高 |

### 2.2.4 什么时候启用 GOAP

不是每个任务都值得启用 scratch_pad。经验法则：

- 单工具、单步骤任务（如 "帮我翻译这句话"）——**不要**启用，徒增 token；
- 多工具规划、明显需要依赖排序的任务——**强烈建议**启用；
- 长链路自主任务（研究类、报告类）——**必须**启用，且配合 `max_depth=8+`。

## 2.3 结构化输出（JSON Mode）：从文本海洋里打捞结构化实体

### 2.3.1 为什么需要 JSON Mode

Agent 的输出常常要写入数据库、传给下游服务，这就要求输出严格符合 schema。仅靠 prompt 要求"请返回 JSON"，现实中约 5% 的概率会多一个逗号或漏一个引号。`jsonmode.py` 的核心思路是：用 Pydantic 定义数据模型 → 自动导出 JSON Schema → 把 schema 塞进 system prompt → 解析时用 Pydantic 做最终校验（失败则重试）。

### 2.3.2 数据提取 Agent 完整例子

```python
from pydantic import BaseModel, Field
from typing import List, Optional
import json

class Person(BaseModel):
    name: str = Field(description="人物全名")
    title: Optional[str] = Field(None, description="职务/头衔")
    organization: Optional[str] = Field(None, description="所属组织")

class Event(BaseModel):
    date: str = Field(description="事件日期 YYYY-MM-DD")
    summary: str
    participants: List[Person]

class ExtractionResult(BaseModel):
    events: List[Event]
    raw_language: str = Field(description="原文主要语言的 ISO 639-1 代码")

SCHEMA = ExtractionResult.model_json_schema()

SYSTEM = f"""你是一个信息提取助手。从用户提供的文本中抽取事件，并严格按以下 JSON Schema 返回结果。

<schema>
{json.dumps(SCHEMA, ensure_ascii=False, indent=2)}
</schema>

只输出一个 JSON 对象，不要任何额外文字或 Markdown 围栏。
"""

def extract(text: str, max_retry: int = 2) -> ExtractionResult:
    messages = [
        {"role": "system", "content": SYSTEM},
        {"role": "user", "content": text},
    ]
    for attempt in range(max_retry + 1):
        resp = client.chat.completions.create(
            model="NousResearch/Hermes-3-Llama-3.1-8B",
            messages=messages,
            temperature=0.1,
            response_format={"type": "json_object"},  # vLLM 支持
        )
        raw = resp.choices[0].message.content
        try:
            return ExtractionResult.model_validate_json(raw)
        except Exception as e:
            # 把错误回注，让模型自修
            messages.append({"role": "assistant", "content": raw})
            messages.append({
                "role": "user",
                "content": f"上一次输出无法通过 schema 校验，错误：{e}。请严格按 schema 重新输出。",
            })
    raise ValueError("JSON extraction failed after retries")
```

三点加分项：

- `response_format={"type": "json_object"}` 是 vLLM 支持的服务端约束，结合 xgrammar / outlines 可做到 100% 合法 JSON；
- Pydantic 的字段 `description` 会被写进 JSON Schema，模型把它当作"字段说明"来理解语义，比光看字段名精确得多；
- 失败重试时把 Pydantic 报错注入对话，模型大概率能自己修复——这比你手写 `try: json.loads` 然后失败放弃要鲁棒得多。

## 2.4 记忆机制：短期上下文与长期 RAG 的双层结构

Agent 如果没有记忆，就是金鱼。Hermes 3 有 128K 的长上下文（Llama 3.1 的遗产），但光靠长上下文不够——成本、延迟、注意力稀释都是问题。工业实践里是短期 + 长期的两层架构：

- 短期记忆：最近 N 轮对话，完整塞进 messages；
- 长期记忆：历史对话 + 外部知识库，切片 embedding 存向量库，按需检索再注入 system prompt。

### 2.4.1 代码骨架

```python
from sentence_transformers import SentenceTransformer
import chromadb

encoder = SentenceTransformer("BAAI/bge-m3")
chroma = chromadb.PersistentClient(path="./memory").get_or_create_collection("kb")

def remember(text: str, meta: dict):
    vec = encoder.encode(text).tolist()
    chroma.add(documents=[text], embeddings=[vec], metadatas=[meta],
               ids=[meta["id"]])

def recall(query: str, k: int = 5) -> list[str]:
    vec = encoder.encode(query).tolist()
    res = chroma.query(query_embeddings=[vec], n_results=k)
    return res["documents"][0]

SHORT_TERM_WINDOW = 10  # 最近 10 轮

def build_messages(user_msg: str, history: list[dict]) -> list[dict]:
    # 1. 检索长期记忆
    snippets = recall(user_msg, k=5)
    memory_block = "\n".join(f"- {s}" for s in snippets)

    system = f"""你是一个有记忆的 AI 助手。

<long_term_memory>
以下是可能与当前对话相关的历史信息（来自向量检索，按相关性排序）：
{memory_block}
</long_term_memory>

请在回答时自然地利用这些信息，不要生硬引用。
"""
    # 2. 短期窗口
    short = history[-SHORT_TERM_WINDOW * 2:]  # 一轮 = user+assistant 两条
    return [{"role": "system", "content": system}, *short,
            {"role": "user", "content": user_msg}]
```

### 2.4.2 工具记忆（Few-shot 注入）

一个特别实用的技巧：把历史上 "问题相似、已验证正确" 的 tool_call 序列作为 few-shot 放进 system prompt。Hermes 对 in-context 示例非常敏感，三五条高质量示例就能把边缘场景的准确率从 70% 拉到 90%+。

## 2.5 Multi-Agent 协作：从单兵到小队

单个 Agent 足够解决 80% 的任务，但剩下那 20%——尤其是研究、写作、审核这类需要不同视角或不同工具集的任务——用多 Agent 协作会显著更优。

### 2.5.1 三种基础模式

#### 模式 A：Router（路由器）

一个主 Agent 只做分诊，把任务甩给专家。

```text
[Router System Prompt]
你是任务路由器。根据用户问题，输出一个 JSON：
{"agent": "<researcher|coder|writer>", "task": "<改写后的子任务>"}
不要回答具体问题。
```

#### 模式 B：Pipeline（流水线）

A → B → C 串行，前一个的输出是下一个的输入。

```text
[Stage A: Researcher]
任务：搜集关于 {topic} 的 5 条高质量资料，输出 JSON 数组 [{url, summary}]

[Stage B: Analyst]
输入：Researcher 的 JSON
任务：提取核心洞察、矛盾点、数据缺口，输出 Markdown

[Stage C: Writer]
输入：Analyst 的 Markdown
任务：改写为 800 字中文科普博文
```

#### 模式 C：Debate（辩论收敛）

多个 Agent 对同一问题独立作答，再由一个 Judge Agent 综合。

```text
[Agent 1 / 2 / 3 System Prompt]
你是独立专家。对用户问题给出你的回答和置信度（0-1）。
不要参考他人意见。

[Judge System Prompt]
你将收到 3 个专家的答案。请指出共识、分歧、各自证据强度，
最终输出综合结论。如果分歧过大，回答"需要更多信息"。
```

### 2.5.2 实战：自动数据研究助手（三 Agent 协作）

```python
from dataclasses import dataclass
from typing import Callable

@dataclass
class Agent:
    name: str
    system: str
    tools: list = None

    def run(self, user_msg: str) -> str:
        messages = [{"role": "system", "content": self.system},
                    {"role": "user", "content": user_msg}]
        if self.tools:
            return run_agent_with_tools(messages, self.tools)  # 复用 2.1 主循环
        return client.chat.completions.create(
            model="NousResearch/Hermes-3-Llama-3.1-8B",
            messages=messages, temperature=0.3,
        ).choices[0].message.content


researcher = Agent(
    name="Researcher",
    system="你负责搜集资料。调用 web_search / fetch_url 工具，"
           "最终输出一个 JSON 数组 [{url, title, summary}]，每条 summary 不超过 100 字。",
    tools=[WEB_SEARCH_TOOL, FETCH_URL_TOOL],
)

analyst = Agent(
    name="Analyst",
    system="你是数据分析师。输入是 JSON 资料数组。"
           "请抽取核心数字、趋势、相互矛盾之处，输出 Markdown 小节 '## 关键洞察'。",
)

writer = Agent(
    name="Writer",
    system="你是技术编辑。把输入的洞察改写为 1500 字中文博文，分 3-5 节，"
           "每节有小标题。保持客观、注明不确定性。",
)


def orchestrate(topic: str) -> str:
    print(f"[Orchestrator] 研究主题：{topic}")
    evidence = researcher.run(f"请研究：{topic}")
    insights = analyst.run(evidence)
    article = writer.run(insights)
    return article


if __name__ == "__main__":
    print(orchestrate("2025 年开源 LLM Agent 框架的格局"))
```

这个骨架非常朴素，但已经体现了 Multi-Agent 的三个工程哲学：

1. 每个 Agent 只知道自己的职责——更短的 system prompt 意味着更专注的注意力；
2. Agent 之间用结构化数据（JSON / Markdown）通信，而非共享对话历史；
3. Orchestrator 是一段普通 Python 代码，不是另一个 Agent——保持控制流的确定性。

### 2.5.3 什么时候别用 Multi-Agent

- 任务延迟敏感：多 Agent 意味着多轮模型调用，延迟成倍增加；
- 任务其实是单一流程：硬拆只会引入信息损耗；
- 你还没把单 Agent 调稳：先让一个 Agent 做对，再谈分工。

## 2.6 错误处理与容错：让 Agent 在真实世界里活下来

生产环境里，模型会"抽风"：输出半截 JSON、调用不存在的工具、参数类型错误、网络超时、API 限流……没有容错的 Agent 活不过一天。

### 2.6.1 JSON 解析失败 → 自修重试

```python
def safe_parse_tool_call(text: str, max_retry: int = 2) -> list[dict]:
    for attempt in range(max_retry + 1):
        try:
            return [json.loads(m) for m in TOOL_CALL_RE.findall(text)]
        except json.JSONDecodeError as e:
            if attempt == max_retry:
                raise
            # 让模型自修
            fix_prompt = f"以下内容存在 JSON 语法错误：{e}\n原文：{text}\n请只输出修复后的 <tool_call> 块。"
            text = client.chat.completions.create(
                model="NousResearch/Hermes-3-Llama-3.1-8B",
                messages=[{"role": "user", "content": fix_prompt}],
                temperature=0,
            ).choices[0].message.content
```

### 2.6.2 未知工具 → 显式拒绝并告知

```python
def execute_tool(call: dict):
    name = call.get("name")
    if name not in LOCAL_FUNCS:
        return {"error": f"未知工具 {name}，可用工具：{list(LOCAL_FUNCS)}"}
    try:
        return LOCAL_FUNCS[name](**call.get("arguments", {}))
    except TypeError as e:
        return {"error": f"参数错误：{e}"}
```

把错误作为 `<tool_response>` 回注给模型，它通常能自己纠正：重新调用正确的工具或修正参数。

### 2.6.3 超时与优雅降级

```python
import asyncio

async def with_timeout(coro, seconds=10, fallback=None):
    try:
        return await asyncio.wait_for(coro, timeout=seconds)
    except asyncio.TimeoutError:
        return fallback or {"error": "tool timeout"}
```

原则：

- 每个 tool 单独限时，而不是整个 Agent 循环限时；
- 超时要把错误回注给模型，让它自己决定放弃还是重试其他路径；
- 对关键工具（如数据库写入）做幂等设计，允许安全重试。

### 2.6.4 可观测性：tool call trace

每次模型输出 + 每次工具调用都要记录，字段建议：`request_id, depth, tool_name, args, result, latency_ms, error`。用 JSON Lines 写文件，事后用 `jq` 能直接做漏斗分析。没有 trace 的 Agent 调试，就像闭着眼修赛车。

## 2.7 部署为 API 服务：vLLM + FastAPI + OpenAI 兼容端点

开发完 Agent，下一步是让你的同事、App、流水线用得上。Hermes 3 的部署首选 **vLLM**，它把 Function Calling 的解析和流式输出都做进了 OpenAI 兼容接口。

### 2.7.1 启动 vLLM 服务

```bash
# 单卡 8B（~16GB 显存，FP16）
vllm serve NousResearch/Hermes-3-Llama-3.1-8B \
    --host 0.0.0.0 --port 8000 \
    --max-model-len 32768 \
    --gpu-memory-utilization 0.92 \
    --enable-auto-tool-choice \
    --tool-call-parser hermes

# 多卡 70B（4 卡 A100）
vllm serve NousResearch/Hermes-3-Llama-3.1-70B \
    --tensor-parallel-size 4 \
    --tool-call-parser hermes \
    --enable-auto-tool-choice \
    --max-model-len 65536
```

关键参数：

- `--tool-call-parser hermes`：启用 vLLM 内置的 Hermes 风格 `<tool_call>` 解析器，响应里直接带结构化 `tool_calls` 字段；
- `--enable-auto-tool-choice`：允许模型自主决定是否调用工具，否则每次必须调；
- `--max-model-len`：按业务需要调，上下文越长 KV cache 越吃显存。

### 2.7.2 FastAPI 封装业务端点

```python
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
from pydantic import BaseModel

app = FastAPI(title="Hermes Agent Service")

class QueryIn(BaseModel):
    message: str
    session_id: str | None = None

class QueryOut(BaseModel):
    answer: str
    trace: list[dict]

@app.post("/agent/query", response_model=QueryOut)
async def agent_query(q: QueryIn):
    answer, trace = await run_agent(q.message, session_id=q.session_id)
    return QueryOut(answer=answer, trace=trace)

@app.post("/agent/stream")
async def agent_stream(q: QueryIn):
    async def gen():
        async for chunk in run_agent_stream(q.message):
            yield f"data: {json.dumps(chunk, ensure_ascii=False)}\n\n"
    return StreamingResponse(gen(), media_type="text/event-stream")
```

### 2.7.3 直接用 OpenAI SDK 调用

既然 vLLM 提供 OpenAI 兼容端点，现有工具链（openai-python、LangChain、LlamaIndex）可以零改造对接：

```python
from openai import OpenAI
client = OpenAI(base_url="http://your-server:8000/v1", api_key="EMPTY")

resp = client.chat.completions.create(
    model="NousResearch/Hermes-3-Llama-3.1-8B",
    messages=[{"role": "user", "content": "AAPL 值得买吗？"}],
    tools=TOOLS,                # OpenAI 风格 tool schema
    tool_choice="auto",
    stream=True,
)
for chunk in resp:
    delta = chunk.choices[0].delta
    if delta.tool_calls:
        print("TOOL:", delta.tool_calls)
    elif delta.content:
        print(delta.content, end="", flush=True)
```

### 2.7.4 流式响应里的推理过程

Hermes 的 `<scratch_pad>` 在流式输出时会逐 token 吐出来。体验上可以把 scratch_pad 渲染成前端的"思考中…"折叠块，把 `<tool_call>` 渲染成工具图标，把最终自然语言吐到主聊天窗口——用户看到的是一个"透明思考"的 Agent，这正是 Hermes 相对于闭源 Agent 的最大体验差异点。

## 小结：从"能跑"到"能干活"的路径图

这一部曲我们走过七个关卡：

1. **多工具编排**——让模型当项目经理，并行拉取异构数据；
2. **GOAP 推理**——用 `<scratch_pad>` 把"思考"显式化、可观测化；
3. **JSON Mode**——用 Pydantic 把模型输出"硬化"成合规结构；
4. **记忆机制**——短期上下文 + 长期 RAG，摆脱金鱼脑；
5. **Multi-Agent**——Router / Pipeline / Debate 三种拓扑应对不同任务；
6. **错误处理**——自修重试、降级、trace 日志，把 Agent 运行在真实的混乱中；
7. **部署**——vLLM + FastAPI + OpenAI 兼容端点，交付给世界。

你可能已经发现：这七件事里没有一件依赖"Hermes 独有的框架"，因为 **Hermes 本质上没有框架**——它是一种让普通 Python 代码加一个原生支持 `<tool_call>` 的模型就能变成 Agent 的哲学。这种极简主义正是它区别于 LangChain、AutoGen、CrewAI 的关键。
