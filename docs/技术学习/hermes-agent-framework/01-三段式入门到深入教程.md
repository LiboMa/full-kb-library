# Hermes Agent 全面教程（三段式）

> 作者：沙漠绿洲（小 zhou）
> 版本对照：Hermes Agent v0.11.0
> 读者定位：从"没听说过 agent"的小白 → 会用 → 懂原理能二次开发

本教程分三段：

- 第一段【入门】：十分钟让它跑起来，学会日常用法

- 第三段【深入】：源码级机制——agent 循环、provider 路由、上下文压缩、Gateway、插件扩展

每一段都能独立读完。读不懂的地方先跳过，不影响后面。

## 第一段【入门】：十分钟上手

### 1.1 Hermes 是什么？一句话说清

Hermes Agent 是 Nous Research 开源的**自主 AI agent 框架**，和 Claude Code、OpenAI Codex CLI、Cursor Agent 是同一类产品：

    你说一句话 → 它调用 LLM（大模型）→ LLM 决定要不要用"工具"
    （比如读文件、跑命令、搜网页）→ 工具执行 → 结果回给 LLM
    → LLM 决定下一步 → ... → 直到任务完成 → 把最终答案给你

和 ChatGPT 网页版最大的区别：**它能直接操作你的电脑**——读写文件、执行命令、搜网页、发消息到微信/Telegram/飞书、定时任务……都是真实发生在你机器上的动作。

### 1.2 安装

一行命令：

    curl -fsSL https://raw.githubusercontent.com/NousResearch/hermes-agent/main/scripts/install.sh | bash

装完第一次跑会进入 setup 向导，让你配：

- 选哪个大模型厂商（OpenAI / Anthropic / OpenRouter / DeepSeek / Gemini 等 20+ 家都支持）
- 填 API key
- 要不要开语音、网页搜索等工具

也可以随时重新配置：

    hermes setup          # 向导
    hermes model          # 只换模型/厂商
    hermes doctor         # 体检，哪里没配好会告诉你

### 1.3 日常三种用法

**① 交互聊天（最常用）**

    hermes

进去就是一个带动画、彩色边框的对话框。直接打字，它会在后台调 LLM 和工具。

**② 一次性问答**

    hermes chat -q "把当前目录下所有 py 文件行数统计一下"

不进交互界面，答完就退出，适合脚本里用。

**③ 后台长任务**

    hermes chat -q "把 ~/myapp 的测试全跑一遍，失败的修一下" &

### 1.4 会话内斜杠命令（/xxx）

进入 `hermes` 后，输入以斜杠开头的命令可以控制会话本身，不走 LLM：

    /new              开新会话（清空上下文）
    /model            切换模型
    /tools            管理要启用哪些工具
    /skill <名字>     加载一个技能（见第二段）
    /rollback         回滚到上一次文件快照
    /compress         手动触发上下文压缩
    /voice on         开启语音对话
    /help             所有命令一览
    /exit             退出

记住 `/new` 和 `/help` 就够了。

### 1.5 第一次实战：三个例子体感一下

**例 1：帮我看代码**

    hermes
    > 读一下 ~/myproj 里的 main.py，告诉我它做什么

它会自己调 `read_file` 工具看文件内容，然后总结。

**例 2：帮我跑命令**

    > 我电脑磁盘哪里占用最多？

它会调 `terminal` 工具跑 `du` 之类的命令，返回结果并解释。遇到危险命令（`rm -rf` 等）默认会停下来问你"是否允许"，这叫**审批机制**。

**例 3：帮我搜网页**

    > 搜一下 Claude 4.7 Opus 的最新评测，写个 300 字总结

它会调 `web_search`（需要配过搜索 provider）拿结果再总结。

到这里你就会用 Hermes 了。

## 第二段【进阶】：把它用成生产力

### 2.1 工具系统（Tools / Toolsets）

Hermes 的一切能力都是**工具**。工具按类别打包成 **toolset**，常用的：

| Toolset        | 干啥                                    |
|----------------|----------------------------------------|
| `terminal`     | 跑 shell 命令、后台进程                   |
| `file`         | 读写、搜索、patch 文件                    |
| `web`          | 网页搜索 + 内容抓取                       |
| `browser`      | 真实浏览器自动化（Playwright/Browserbase）|
| `code_execution`| 沙箱跑 Python 代码                       |
| `vision`       | 看图                                    |
| `image_gen`    | 画图                                    |
| `tts`          | 文字转语音                               |
| `memory`       | 持久化记忆（跨 session）                  |
| `session_search`| 搜过去的对话历史                          |
| `delegation`   | 派生子 agent 并行干活                     |
| `cronjob`      | 创建/管理定时任务                         |
| `messaging`    | 跨平台发消息（Telegram/飞书/微信等）       |
| `todo`         | 会话内 TODO 跟踪                         |

管理方式：

    hermes tools              # 交互式 TUI 勾选
    hermes tools list         # 看全部 + 启用状态
    hermes tools enable web
    hermes tools disable image_gen

⚠️ 工具变更**在下个会话生效**（为了保 prompt cache 命中率）。改完记得 `/new` 或重开。

### 2.2 Skills（技能）—— Hermes 的灵魂

技能是"**过程记忆**"——一段 Markdown 写的可复用做事方法。比如：
- "怎么通过 gh CLI 做 PR review"
- "怎么给茅台写周报"
- "怎么调用飞书文档 API"

技能本质是一个 `SKILL.md` 文件 + 可选的 `references/`、`templates/`、`scripts/`。每次 agent 看到相关任务，会自动加载对应技能，照着执行。

浏览与安装：

    hermes skills browse              # 浏览 hub 上所有技能
    hermes skills search "github"
    hermes skills install github-pr-workflow
    hermes skills list                # 看已装的
    hermes skills update              # 检查更新

会话内临时加载：

    /skill requesting-code-review

命令行预加载：

    hermes -s plan -s writing-plans

**自己写技能**（5 分钟）：

    mkdir -p ~/.hermes/skills/my-skill
    cat > ~/.hermes/skills/my-skill/SKILL.md <<'EOF'
    ---
    name: my-skill
    description: 我的私人写作风格规范
    ---
    # My Skill
    当用户让我写文章时：
    1. 先列提纲
    2. 每段不超过 3 句
    3. 结尾给行动清单
    EOF

下次 agent 就会发现并使用它。或者让 Hermes 自己帮你写："我刚才做的这个流程挺好用的，帮我存成技能。"它会调 `skill_manage` 工具直接创建。

### 2.3 Memory（长期记忆）

两种 store：

- **user**：你是谁，偏好、习惯、昵称、时区、风格
- **memory**：环境事实、项目约定、工具坑、踩过的经验

Hermes 每次会话启动时自动注入。你可以显式命令：

    > 记住：我叫老周，在上海，偏好简洁回答
    > 别再用 emoji 了

它会调 `memory` 工具写入。下次新会话（即使明天）它还记得。

查看和管理：

    ls ~/.hermes/memory/

后端可换：内置、Honcho、Mem0、Supermemory（看 `hermes memory setup`）。

### 2.4 Session（会话）持久化

所有对话都存在 SQLite（`~/.hermes/sessions/`），可搜可恢复：

    hermes sessions list
    hermes sessions browse          # 交互选
    hermes --continue               # 恢复最近一次
    hermes --resume <session-id>
    hermes sessions export out.jsonl

会话内全文搜索：

    > 我们上次讨论茅台估值的时候你提到的几个指标是啥？

它会调 `session_search` 在所有历史会话里检索。

### 2.5 Cron（定时任务）

让 Hermes 定期自动跑任务，结果自动发回给你。

    hermes cron create "0 9 * * 1" \
      --prompt "拉取 600519 本周行情，写成中文周报"

    hermes cron list
    hermes cron run <job-id>       # 立即触发一次
    hermes cron pause <job-id>

时间格式支持：`30m`、`every 2h`、标准 cron `0 9 * * *`、ISO 时间戳。

交付目的地默认是当前聊天。也能配成发到 Telegram、飞书、邮件。

### 2.6 Gateway（跨平台部署）

**这是 Hermes 最独特的能力之一**：同一个 agent 可以同时活在你的：

- Telegram、Discord、Slack、WhatsApp、Signal、Matrix
- 飞书、企微、钉钉、微信（BlueBubbles / Weixin）
- 邮件、短信
- Home Assistant（智能家居）
- 自建 API Server、Webhook

设置：

    hermes gateway setup          # 配各平台 token
    hermes gateway install        # 装成系统服务
    hermes gateway start
    hermes gateway status

之后你在任何平台 @ 它或 DM，它都能直接用全套工具干活——不只是聊天，还能读你服务器的文件、跑命令、发邮件等。

### 2.7 Profiles（多实例隔离）

一套机器上跑多个独立 Hermes——工作一个、个人一个、实验一个，配置/会话/记忆/技能全部隔离：

    hermes profile create work --clone default
    hermes profile use work
    hermes                        # 现在跑的是 work profile

### 2.8 子 agent 并行（delegate_task）

会话内让 agent 自己派子任务：

    > 并行研究：A) GRPO 最新论文  B) DPO 最新论文
    > 各自写一页总结

Hermes 会调 `delegate_task` 工具，派 2 个独立子 agent 同时跑，只把最终摘要汇总回主会话——**大量中间 tool output 不污染主上下文**。

### 2.9 MCP（Model Context Protocol）

MCP 是 Anthropic 提的工具协议标准。Hermes 既能当 **MCP 客户端**（接入外部 MCP server，比如 GitHub、Filesystem、PostgreSQL 的官方 MCP），也能当 **MCP server**（被 Claude Desktop / Cursor 调用）：

    hermes mcp add github --command "npx @modelcontextprotocol/server-github"
    hermes mcp list
    hermes mcp serve              # 把 Hermes 暴露成 MCP server

到这里你已经是高级用户了。

## 第三段【深入】：源码级机制

> 这一段帮你理解 Hermes **怎么工作的**，方便排错、二次开发、写插件。

### 3.1 总体架构

### 3.2 Agent 主循环（run_agent.py）

**核心是同步循环**（不是 asyncio）。简化伪代码：

    while api_call_count < max_iterations and budget.remaining > 0:
        if interrupt_requested: break

        response = llm_client.chat.completions.create(
            model=model,
            messages=messages,
            tools=tool_schemas,
        )

        if response.tool_calls:
            for call in response.tool_calls:
                result = handle_function_call(call.name, call.args, task_id)
                messages.append(tool_result_message(result))
            api_call_count += 1
            continue

        # 没 tool_calls，只有文本 → 任务完成
        return response.content

几个关键细节：

**消息格式统一用 OpenAI chat.completions 格式**：`{role: system|user|assistant|tool, content, tool_calls?, tool_call_id?}`。这样所有 provider 都能复用同一套消息管理逻辑，各 provider 适配器只负责转成该家 API 需要的形状。

**Budget 机制**（`iteration_budget`）：除了最大迭代次数，还有 token 预算。快用完时会触发 `_budget_grace_call`——给 agent 最后一次机会收尾。

**中断机制**：用户按 Ctrl+C 触发 `_interrupt_requested` 标志，循环在下一次检查时优雅退出，保存已有消息。

**Prompt Caching**：Anthropic 和 OpenAI 都支持，系统提示 + 历史前缀做成静态片段缓存。所以 Hermes 有个铁律：**对话中途不改 system prompt、不改 tool schema**——改了缓存就失效。这也是 "启用/禁用工具要等 /new 之后生效" 的根本原因。

### 3.3 Provider 适配层（agent/*_adapter.py）

不同厂商的 API 形状不一样：

- OpenAI 风格（`chat.completions`）：多数 provider 兼容
- Anthropic 原生：`messages` 格式不一样，system 单独放
- Bedrock：AWS 签名 + Anthropic 消息格式
- Gemini：tools 叫 `functionDeclarations`，消息角色不同
- Codex Responses：OpenAI 新的 reasoning API 格式

每个都有独立适配器：

    agent/anthropic_adapter.py
    agent/bedrock_adapter.py
    agent/gemini_native_adapter.py
    agent/codex_responses_adapter.py
    ...

适配器职责：

1. **出方向**：把 Hermes 内部 OpenAI-format messages + tool schemas 转成该家 API
2. **入方向**：把该家响应转回 OpenAI-format（tool_calls、reasoning、usage）
3. **错误分类**（`error_classifier.py`）：把各家不同错误归一到"速率限制/上下文超长/鉴权失败/服务故障"几类

**Credential Pool**（`credential_pool.py`）：同一 provider 可配多个 API key，用完/被限速自动轮换。

**Rate Guard**（`nous_rate_guard.py`）：Nous 官方 endpoint 的客户端限流。

### 3.4 工具注册与分发（tools/registry.py + model_tools.py）

工具**自动发现**，不需要手动维护列表。流程：

1. `tools/` 目录下每个 `.py` 文件，在 `import` 时执行 `registry.register(...)` 把自己登记进中央 registry
2. `model_tools.py` 的 `discover_builtin_tools()` 触发整个 `tools/` 包的 import
3. 启动时根据用户启用的 toolset 过滤，把对应工具的 schema 打包进 LLM 请求
4. LLM 返回 `tool_calls` → `handle_function_call(name, args)` 查 registry 找 handler → 执行 → 返回 JSON 字符串

新增工具 = 加一个文件 + `registry.register()`，不改其他代码：

    # tools/my_tool.py
    from tools.registry import registry

    def check_requirements() -> bool:
        return bool(os.getenv("MY_API_KEY"))

    def my_tool(param: str, task_id: str = None) -> str:
        return json.dumps({"success": True, "data": "..."})

    registry.register(
        name="my_tool",
        toolset="mine",
        schema={"name": "my_tool", "description": "...", "parameters": {...}},
        handler=lambda args, **kw: my_tool(args.get("param"), kw.get("task_id")),
        check_fn=check_requirements,
        requires_env=["MY_API_KEY"],
    )

`check_fn` 不通过时工具自动隐藏——LLM 根本看不到它，不会误调。

### 3.5 终端后端（tools/environments/）

`terminal` 工具不是直接 `subprocess.run`——它有**可插拔后端**：

- `local`：本地进程（默认）
- `docker`：每个命令跑在独立容器里
- `ssh`：跑在远程机器上
- `modal`：Modal.com 云沙箱
- `daytona` / `singularity`：其他沙箱

切换：`hermes config set terminal.backend docker`。

所有后端实现同一接口：同步 foreground、后台进程（带 session_id + poll/wait/kill）、PTY 模式（给 `vim` / `claude` 这种交互 CLI 用）。

### 3.6 上下文压缩（agent/context_compressor.py）

上下文快满时自动触发（默认 token 数到 50% 时）。压缩策略：

1. **保留**：system prompt、最近 N 轮、所有 tool_call ↔ tool_result 对（不能打断）
2. **摘要**：中段历史用辅助 LLM 总结成一段 `<compressed_history>`
3. **目标**：压到 `target_ratio`（默认 20%）

用户可手动触发 `/compress`，也能看到 `compress_ratio` 等统计。

**Context Engine** 插件（`plugins/context_engine/`）允许你换掉整套压缩策略。

### 3.7 记忆系统（agent/memory_manager.py + plugins/memory/）

启动时：

1. 读取 `~/.hermes/memory/memory.md` 和 `user.md`
2. 注入到 system prompt
3. 运行时 `memory` 工具 add/replace/remove → 直接改这两个文件

可换后端（`plugins/memory/` 下每个目录一个 provider）：

- **built-in**（默认）：纯 Markdown 文件
- **Honcho**：带 embedding 的语义记忆
- **Mem0**：LLM 驱动的记忆图谱
- **Supermemory**：托管服务

切换：`hermes memory setup`。

### 3.8 Session 存储（hermes_state.py）

SQLite + FTS5：

- 每条消息是一行（role、content、timestamp、tool_calls JSON）
- FTS5 虚表索引 content 字段 → `session_search` 工具做关键词/布尔查询
- 会话 ID 是时间戳 + 随机后缀
- Profile 隔离：每个 profile 有独立 DB 文件

### 3.9 Gateway 架构（gateway/）

    gateway/run.py            主循环 + 路由 + 命令分发
    gateway/session.py        每个 chat/user 映射到一个持久 session
    gateway/config.py         解析平台配置
    gateway/platforms/        每平台一个 adapter
    gateway/builtin_hooks/    系统级钩子（boot message 等）

每个平台 adapter 实现：

- `connect()` / `disconnect()`
- `send_message(target, content)` / `send_media()`
- 监听事件流（新消息、reaction、语音等）→ 转成内部 `GatewayEvent` 丢给 run.py
- home channel 管理

**会话路由**：`(platform, chat_id, thread_id)` → 一个持久 session。同一用户同一对话的历史连续，跨 chat 隔离。

**审批流**：危险命令在 gateway 里会触发 `approval_pending`，把按钮/命令 `/approve` `/deny` 发给用户。

### 3.10 插件系统（plugins/）

Hermes 允许在不 fork 主仓库的前提下扩展：

    plugins/
    ├── memory/           # 记忆后端
    ├── context_engine/   # 上下文压缩策略
    ├── dashboard/        # Web 监控面板
    ├── image_gen/        # 额外图像模型
    └── <你自己写的>/      # 放进去就能被发现

每个插件有 `plugin.toml` 描述元数据 + Python 入口。

### 3.11 斜杠命令中央注册（hermes_cli/commands.py）

所有 `/xxx` 命令定义在一个 `COMMAND_REGISTRY: list[CommandDef]` 里。下游消费者全部从它派生：

- CLI 分发：`resolve_command()`
- Telegram BotCommand 菜单
- Slack `/hermes` 子命令路由
- 自动补全（prompt_toolkit）
- `/help` 输出
- Gateway 钩子识别

所以加一个新命令 = 在 registry 里加一个 `CommandDef`，其它地方全部自动同步。

### 3.12 ACP 适配（IDE 集成）

`acp_adapter/` 实现 Zed / Cursor / VS Code 的 Agent Client Protocol，让 Hermes 嵌进编辑器当 AI agent。也支持反向——Hermes 调用外部 ACP agent（比如 `claude --acp --stdio` 把 Claude Code 当子 agent）。

### 3.13 调试建议

遇到问题的排查顺序：

    hermes doctor                                # 体检
    tail -f ~/.hermes/logs/agent.log             # agent 日志
    tail -f ~/.hermes/logs/errors.log            # 只看错误
    tail -f ~/.hermes/logs/gateway.log           # gateway 日志
    hermes config                                # 看当前配置
    journalctl --user -u hermes-gateway -n 60    # gateway 系统日志

常见陷阱：

- **工具没出现**：没启用 toolset、缺 env var、或者在旧 session（要 `/new`）
- **Edge TTS 中文失败**：voice 锁定英文，改成 `zh-CN-XiaoxiaoNeural`
- **Gateway 启动崩**：`config.yaml` 里 `home_channel` 写成裸字符串（要是 dict）
- **Copilot 403**：必须用 `hermes login` 的专用 OAuth，不能复用 `gh auth`

### 3.14 二次开发入口

想改 Hermes：

    git clone https://github.com/NousResearch/hermes-agent
    cd hermes-agent
    python -m venv venv && source venv/bin/activate
    pip install -e '.[dev]'
    ./scripts/run_tests.sh         # ~15k 测试

四个最常碰的地方：

1. **新工具**：加 `tools/xxx.py` + `registry.register()`，再把 toolset 名加进 `toolsets.py` 的 `_HERMES_CORE_TOOLS`
2. **新斜杠命令**：`hermes_cli/commands.py` 里加 `CommandDef`，`cli.py::process_command()` 里加 handler
3. **新 provider**：写 `agent/xxx_adapter.py`，在 provider 工厂里登记
4. **新平台**：`gateway/platforms/xxx.py`，见 `ADDING_A_PLATFORM.md`

## 学习路径建议

- **第 1 天**：按第一段跑起来，每天强迫自己用 `hermes` 代替一次 ChatGPT
- **第 1 周**：装 3-5 个技能（`hermes skills browse`），感受技能如何改变回答质量
- **第 2 周**：开一个定时任务 + 接通一个消息平台（Telegram 最简单）
- **第 1 个月**：写自己的第一个 skill，让 Hermes 适配你的工作流
- **2 个月+**：读 `run_agent.py` 主循环 + 一个 adapter 源码，尝试加一个自己的工具
